# MokaWeb - Frontend

The application represents the user-facing side of the MokaWeb ecosystem.

The project was developed as the natural counterpart of the MokaWeb backend and provides a complete interface for managing employees, leave requests, scheduling templates, and weekly work schedules. Through the frontend, managers can organize staff, manage employee availability and leave requests, generate weekly schedules, and publish shifts, while employees can view their schedules and submit leave requests.

The application was developed as a prototype intended to be used within the company where I currently work, with the goal of validating the system's usability and evaluating its effectiveness in a real operational environment.

While the backend focuses on business rules, scheduling logic, security, and data persistence, the frontend acts as the direct point of interaction with users, providing an intuitive, responsive, and user-oriented experience.

The objective of the project is to transform a collection of REST APIs into a practical tool that can be used by both managers and employees in their daily activities. Particular attention was given to application navigation, role-based access control, centralized state management, and the separation between presentation logic and backend communication.

# Overview

MokaWeb Frontend is a Single Page Application developed using React, TypeScript, Redux Toolkit, Axios, and React Router.

The application communicates exclusively through the REST APIs exposed by the MokaWeb backend and provides all the functionality required by the system:

* authentication and role-based access control;
* employee management;
* availability management;
* leave request management;
* user account management;
* schedule template management;
* weekly schedule generation and publication;
* published schedule visualization.

The application supports three different user roles, each characterized by specific responsibilities and permissions:

* ADMIN: responsible for creating and managing manager accounts.
* MANAGER: responsible for employee management, availability management, leave requests, schedule template creation, and schedule generation.
* STAFF: allowed to view published schedules and submit leave requests.

The frontend does not directly implement the business rules of the system. Instead, it acts as the presentation and interaction layer, delegating data validation, authorization, and scheduling logic to the backend services.


# Architecture

The application follows a feature-based architecture, where each functional area of the system owns its own components, services, types, and pages.

Instead of organizing the code exclusively by technical categories, each feature is maintained within a dedicated module that contains all the elements required for its functionality.

The main application modules include:

* authentication;
* employee management;
* leave requests;
* staff account management;
* manager account management;
* scheduling templates;
* weekly schedules.

The architecture is organized around the following layers:

* Pages: route-level screens responsible for handling user interactions.
* Components: reusable UI elements and presentation components.
* API Services: the layer responsible for HTTP communication with the backend through Axios.
* Redux Store: centralizes the application's global state management.
* Types: define shared TypeScript models and keep the frontend aligned with backend DTOs.

This structure reduces coupling between different areas of the application, improves maintainability, and allows individual features to evolve independently without requiring invasive changes throughout the rest of the system.

# Authentication & Authorization

Application authentication is based on JWT (JSON Web Token) access tokens generated by the backend.

When a user successfully logs in, the following process takes place:

1. credentials are sent to the backend;
2. the backend validates the credentials and returns a JWT access token;
3. the token is stored locally;
4. the frontend retrieves the authenticated user information through the /auth/me endpoint;
5. user information is stored inside the global Redux state.

All subsequent API requests automatically include the token through a dedicated Axios interceptor, which injects the authorization header into every protected request.

Protected pages are managed through the ProtectedRoute component, which verifies:

* authentication status;
* application loading state;
* roles authorized to access the requested resource.

Although route protection improves the user experience and prevents navigation to unauthorized pages, actual system security is enforced by the backend through Spring Security and a Role-Based Access Control (RBAC) model.

# State Management

Global application state is managed through Redux Toolkit.

Within the project, Redux is primarily used to centralize authentication-related information and avoid manually propagating data through the component hierarchy.

The store maintains information such as:

* access token;
* authenticated user;
* loading state;
* error state.

The authentication workflow is implemented through Async Thunks, which allow asynchronous operations to be handled while keeping business logic separated from React components.

The main thunks used by the application are:

* `login`
* `loadCurrentUser`
* `updateCurrentUserProfileImage`
* `uploadCurrentUserProfileImage`

Each asynchronous operation follows the standard Redux Toolkit lifecycle:

* pending
* fulfilled
* rejected

This approach centralizes state transitions, makes application behavior during API requests predictable, and allows React components to focus primarily on data presentation and user interaction.

# Routing & Navigation

Application navigation is handled through React Router.

The routing system is organized to clearly separate the different areas of the application according to the user's authentication status and role:

* public routes;
* authenticated routes;
* manager routes;
* staff routes;
* administrator routes.

Each protected route is wrapped by the ProtectedRoute component, which verifies authentication status and authorized roles before rendering the requested page.

This approach allows client-side navigation control while presenting each user only with the functionality appropriate to their application role.

The shared AppLayout component provides common interface elements such as the navigation bar and the overall page structure, ensuring a consistent navigation experience throughout the application.

# Backend Integration

Communication with the backend is centralized through a dedicated Axios client responsible for configuring HTTP requests and automatically injecting the JWT token into protected requests through an interceptor.

The application also uses a set of specialized services that isolate communication logic from the rest of the user interface.

The main services include:

* `authService`
* `employeeService`
* `leaveRequestService`
* `templateService`
* `weeklyScheduleService`

React components do not communicate directly with REST endpoints. Instead, they use these services as an intermediate layer. This keeps presentation logic separated from communication logic and allows API-related changes to be managed in a single place.

This separation reduces coupling between the user interface and the backend, improves maintainability, and makes future system evolution easier.


# TypeScript & Domain Consistency

Particular attention was given to maintaining consistency between the backend domain model and the way data is represented within the frontend.

Whenever the backend exposes enumerations or restricted sets of values, the frontend uses TypeScript Union Types to represent those values in a safe and controlled manner.

Examples include:

* `Role`
* `EmployeePriority`
* `EmployeeSkillName`
* `EmployeeProficiency`
* `DayOfWeekValue`
* `LeaveRequestStatus`

On the other hand, values representing serialized backend data such as dates, times, and timestamps are intentionally kept as simple strings, since they directly reflect the format returned by the REST APIs.

This approach improves type safety, reduces the risk of invalid values being used throughout the application, and keeps the frontend aligned with the DTOs and domain rules defined by the backend.
